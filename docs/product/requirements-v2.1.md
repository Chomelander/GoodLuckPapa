## 一、产品概述

### 1.1 定位

基于多流派教育理论、依托 AI 能力（P1 阶段引入）的**儿童发展顾问系统**。为 0-6 岁孩子的父母提供：

- 科学的发育评估与里程碑对照
- 结构化的家庭观察日志（蒙氏观察法）
- 个性化的活动推荐与执行引导
- 兴趣点的持续识别与培养路径
- 阶段性成长复盘（周/月/季）

**核心差异**：不是打卡工具，是一个帮父母**学会观察孩子**的方法论载体，AI 在 P1 阶段进一步将其升级为"持续学习你的孩子的顾问"。

### 1.2 核心理念

| 理念 | 说明 |
|------|------|
| 科学底座 | 蒙台梭利 + 皮亚杰 + 多元智能 + RIE + 正面管教 + 瑞吉欧 + 华德福组合 |
| 闭环进化 | 观察 → 记录 → 评估 → 调整环境 → 再观察，每个环节有工具支撑 |
| 尊重个体 | 基于"你的孩子"的实际数据，不与他人比较，只与自己上一阶段比 |
| 低摩擦 | 每次活动主动记录 ≤ 5 分钟，融入日常陪伴 |

### 1.3 产品形态

| 端 | 形态 | MVP 状态 |
|----|------|---------|
| 手机端 | PWA（安装到桌面）/ H5 | 主要开发目标 |
| 服务端 | API 服务（P1 引入） | MVP 不实现 |

### 1.4 UI 参考

视觉设计以 `prototype.html` 为基准参考，新起代码开发（不直接修改原型文件）。原型中定义的颜色系统、组件风格、交互模式均应还原。

### 2.2 维度框架（三层映射）

本 App 采用**双层前台 + 专业后台**的维度架构：

#### 前台展示层（父母感知）：5 维度

| 维度代码 | 名称 | 理论来源 | 色值 |
|---------|------|---------|------|
| `cognitive` | 认知 | 皮亚杰 + Gardner 逻辑/空间智能 | `#6B9BDF` |
| `language` | 语言 | Gardner 语言智能 + 蒙氏语言敏感期 | `#9B7FD4` |
| `motor` | 运动 | Gardner 身体动觉 + RIE 运动发展 | `#5EBD8A` |
| `emotional` | 情感 | Gardner 人际/内省 + 正面管教 | `#E8956D` |
| `habit` | 习惯 | 华德福节律 + 正面管教 + 蒙氏秩序敏感期 | `#C4A830` |

#### 后台活动层（活动编号体系）：H 编号

格式：`{领域前缀}-{最小月龄}-{序号}`

| 前缀 | 领域（蒙氏） | 对应前台维度 |
|------|------------|------------|
| P | 日常生活 Practical Life | `habit` |
| S | 感官教育 Sensorial | `cognitive` |
| M | 运动 Movement | `motor` |
| L | 语言 Language | `language` |
| E | 社会情绪 Emotional/Social | `emotional` |
| C | 文化 Cultural（3岁+） | `cognitive` |
| N | 数学 Numeracy（3岁+） | `cognitive` |

**示例**：`S-6-04` = 感官类 / 6月起 / 第4号 = 物体永久性盒

#### 评估分析层（月度复盘）：蒙氏 10 内在维度

仅在月度复盘雷达图中展示，日常不展示给父母：

独立能力、秩序感、专注力、手眼协调、感官敏锐度、逻辑思维、语言表达、社会情绪、自我纠错、内在动机

每维度三级判定：**萌芽 / 发展中 / 稳定表现**（详见 `02_核心框架与模板.md` E 部分）

### 2.3 蒙氏敏感期 × 活动推荐

| 敏感期 | 年龄范围 | 系统行为（MVP）|
|--------|---------|--------------|
| 秩序敏感期 | 0-3 岁 | 推荐固定流程的活动；提醒父母维持环境一致性 |
| 感官敏感期 | 0-5 岁 | 推荐多感官刺激活动；按月龄递进复杂度 |
| 语言敏感期 | 0-6 岁 | 语言维度活动按月龄密度调整 |
| 运动敏感期 | 0-4 岁 | 大运动→精细动作递进；结合 RIE 自由运动原则 |

### M4. 观察记录（精简4字段，坚持优先）

**设计原则**：完成率 > 完整度。4个字段父母 30 秒内完成，系统从已有数据自动推算其余指标。

**触发方式**：
- 停止计时后自动弹出（最常用路径）
- 成长页「+」FAB 按钮手动新建

**4个核心字段**：

| 字段 | 控件 | 用途 |
|------|------|------|
| 发起方式 | 3选1按钮：自己来的 / 我邀请的 / 我带着做的 | F规则·自主选择频率 |
| 专注时长 | 秒表自动填入（或手动输 MM:SS） | F规则·投入深度；被动采集，父母不需主动计数 |
| 情绪状态 | 3选1按钮：全程平稳 / 有波动但ok / 需要帮他 | 活动难度匹配判断 |
| 有什么特别的？ | 自由文本，选填 | 覆盖拓展行为+困难+亮点 |

**自动推算字段**（父母不填，系统计算）：
- `dailyCount`：同一天同一活动的记录条数，DB 统计
- 连续天数：从记录时间序列自动计算，用于 F 规则「连续3天」判定
- 拓展行为：「有什么特别的？」字段不为空 → 视为有拓展行为

**废弃字段**（从必填移除）：
完成度、自主纠错次数、重复操作次数、遇到的困难、应对方式——这些字段依赖父母实时计数，实际场景无法保证，改为通过「有什么特别的？」文本覆盖。

**【观察眼训练】placeholder 引导**（v1.2新增）：

「有什么特别的？」文本框 placeholder 分4类轮换，每次打开随机选1条，引导父母写行为事实而非情绪判断：

| 类型 | Placeholder 示例 |
|------|----------------|
| 拓展行为 | 「他有没有做什么出乎意料的事？比如：把球放到盒盖上，而不是放进孔里」 |
| 迁移行为 | 「今天在其他地方有没有看到类似的动作？比如：洗澡时用杯子舀水」 |
| 坚持行为 | 「遇到做不到的时候他怎么反应？继续试了几次，还是换了个方法？」 |
| 重复行为 | 「他有没有反复做同一个动作？大概做了几次？」 |

文本框下方固定灰色小字：「写他做了什么，不用评价好不好」

**observeAnchor 填空引导**（v1.8新增）：

记录表单在「有什么特别的？」文本框上方，展示来自该活动 `observeAnchor` 字段的**逐条填空问题**：

```
观察锚点
  球消失后，他有没有用手去找？
  [ _________________ ]（输入框，placeholder：记录你的观察…）

  他拿球时，有没有先看圆孔再对准放入？
  [ _________________ ]

有什么特别的？（自由补充）
  [ _________________ ]
```

- `observeAnchor` 以中文「？」为分隔符拆分（`parseAnchorQuestions`），每个问题独立显示为 label + text input
- 提交时，**有填写的问题**合并为「问题 → 答案」格式，追加至 note（`mergeAnchorAnswers`），中间用 `---` 分隔自由文本
- **编辑已有记录时**（M_RecordEdit 进入编辑态）：**不展示**填空区（已有合并内容直接显示在文本框中）
- 活动无 `observeAnchor` 字段时：直接展示「有什么特别的？」文本框，行为与 v1.7 相同

**提交**：写入 `records` store，自动填写 `date`、`ageMonths`。

**秒表组件**：从活动卡「开始计时」触发，自动运行，停止时值写入 focusSec，并弹出记录表单。

### M4.5 正向会话结束卡片（v1.6新增）

**设计原则**：每次观察都有意义。在数据反馈后给予**情绪上的结束与肯定**，让父母获得自然的会话结束点。

**触发时机**：M4 记录表单提交成功后，出现 1 次（不重复）

**UI 形式**：底部弹出卡片（不阻断，可关闭，3秒后自动消失）

**内容逻辑**：

```
根据本次记录的 focusSec、initType、emotion 三字段，动态生成反馈：

基础反馈：
  「今天的观察很有意义」
  
客观成就盘点（择一或多个）：
  ✓ 他专注了 X 分钟（如果 > typicalSec，加注「比这个月龄典型长」）
  ✓ 他自己主动做的（如果 initType === 'self'）
  ✓ 他探索了新的方式（如果 note 非空）
  ✓ 他在练习坚持（如果 emotion === 'wave-ok'）

结语（非评价）：
  「继续下一个活动很自然。」
  或 「今天就这样吧。」

按钮：
  [好的，结束了] → 关闭卡片，返回首页
  [继续做] → 关闭卡片，建议今日推荐的下一个活动
```

**文案原则**：
- 不出现「做得很棒」「真厉害」等评价（避免成人中心化）
- 而是陈述客观事实（他花了多久时间、他自己选择的、他试了新方式）
- 强调**观察过程本身的价值**，而不是结果

**示例**（S-6-04 物体永久性盒，8 个月，12 分钟，自主，平稳）：

```
今天的观察很有意义

✓ 他专注了 12 分钟
  （比这个月龄典型长）
```

### M6.5 情境化正向提示系统

**设计原则**：蒙氏理念的克制要求违反养育本能，强制禁令会让父母焦虑。改为在**对的时机**出现**一次**支持型提示，允许自然，不设禁令。

**触发规则**：每条提示在该情境下**只出现1次**（通过 LocalStorage 记录已展示），之后永不重复，不打扰。

| 触发节点 | 提示文案 | 语气 |
|---------|---------|------|
| **首次进入活动详情** | 「小贴士：示范一遍后退开，让他自己探索——哪怕做"错"了也没关系，那是他的发现过程」 | 支持型 |
| **首次停止计时（活动结束）** | 「记录完成！他重复多少次都正常，重复就是他在学习，不用担心他"卡住了"」 | 安心型 |
| **同一活动连续3天记录** | 「他对这个很感兴趣，继续放着就好，暂时不用换新的」 | 肯定型 |
| **首次标记里程碑达成** | 「太好了！记录下来就够了，不用特别告诉他"你真棒"——这是他自己做到的」 | 引导型 |
| **记录中「发起方式」选「我带着做的」连续2次** | 「偶尔带着做完全没问题。如果想试试让他自己来，可以把材料放好后退一步，看他怎么做」 | 无压力型 |
| **首次完成周复盘** | 「数据是帮你看见他的，不是用来给他打分的。只和他自己上周比，就够了」 | 解压型 |

**设计细节**：
- 提示以**底部小卡片**形式出现（非弹窗，不阻断操作），3秒后可自动消失，也可手动关闭
- 提示出现时机在操作**完成之后**，不在操作之前（不设门槛）
- 不出现「禁止」「不能」等负向词汇
- 全部提示可在设置页「育儿小贴士」中回顾

### M7. 兴趣点识别（F规则，本地计算）

**触发时机**：每次提交记录后，后台自动计算；周复盘时汇总展示。

**计算逻辑**（依据 02-F 快速判定版，v1.2 修订）：

对每个 `actId`，从 `records` store 中取最近 7 天记录，计算以下 5 项：

| 判定项 | 计算规则 | 数据来源 |
|--------|---------|---------|
| 单日主动重复 ≥ 3 次 | 同一天同一 actId 的记录条数 ≥ 3 | 自动统计（dailyCount 替代字段）|
| 专注超典型 50% | `focusSec > typicalSec[ageMonths] * 1.5` | 秒表采集 |
| 连续 3 天主动选择 | 连续 3 天 `initType === 'self'` | 记录时间序列 |
| 出现拓展/迁移行为 | `note !== ''`（补丁A的迁移确认字段 = true 时权重更高） | note 字段 |
| 遇挑战不放弃 | 暂无精确对应字段，由 note 语义推断（P1 AI处理）| 占位，MVP 不计入 |

**【F规则锚点机制】**（v1.3修订）：

判定条件：**第2项（专注时长）必须满足** + 其余3项中任意满足1项 → 标注「兴趣候选」

> 锚点理由：专注时长由秒表被动采集，是4项中唯一可靠的客观数据。其余3项依赖父母主动记录，可靠性较低。以专注时长为锚点，防止系统因父母偶尔填写note就误判兴趣。

**数据基础标注**：兴趣候选角标旁显示小字「基于X条记录」，有效记录 < 3条时角标颜色降级（灰色），提示父母数据尚不充分。

**【补丁A：自主行为独立入口】**（v1.2新增）

独立于推荐活动的「今天他自己做了什么？」入口，位于今日页活动卡下方：

```
今天他自己做了什么？（选填）
[随便写，比如：自己爬去拿了那个木盒子]
这个动作在别处也出现了吗？  ○ 有  ○ 没有
```

- 提交后生成一条 `actId = null` 的特殊记录，归入「自主观察」类别
- 「别处也出现」= 迁移行为确认，在 F 规则第4项中权重 ×2
- 每天仅需记录1次，不强制，入口常驻

**【补丁C：兴趣候选4周追踪时间线】**（v1.2新增）

兴趣候选标注后，系统自动开启4周追踪：

```
成长页 · 兴趣 Tab
┌────────────────────────────────────┐
│ 兴趣候选 · 感官容器类               │
│ 首次识别：4月3日                    │
│ ████████░░░░░░░░░░  第2周/共4周    │
│                                    │
│ 下一步：继续观察，4周后确认          │
│ 如持续，建议增加：套杯/叠叠乐/形状盒 │
└────────────────────────────────────┘
```

4周到期时弹出确认卡：「这个兴趣还在吗？」→ 父母确认 → 升级为「候选优势」，系统给出具体材料建议（来自 G.3 映射表）。

**展示**：
- 活动卡右上角「兴趣候选」角标
- 成长页「兴趣」子 Tab：追踪时间线列表

### M_ObsGuide. 观察引导系统（4层闭环，v1.4新增）

**设计原则**：观察是蒙氏方法的核心，但「观察什么」对小白家长不是自明的。4层系统在不干扰活动流程的前提下，在正确时机给出正确引导。全部引导内容来自活动对象的结构化字段，不依赖 AI，本地计算。

**4层架构总览**：

```
第1层 · 开始前    显示 observeFocus[]（2-3条具体行为焦点 + 发育意义）
第2层 · 计时中    显示 observeAnchor（单句锚点，≤25字，极简）
第3层 · 提交后    即时数据反馈卡（focusSec vs 月龄典型 + initType 解读）
第4层 · 下次前    活动卡顶部显示 currentPhase → phases[].tip
```

---

**第1层：开始前观察焦点**

触发时机：活动详情页展开「活动小贴士」后，步骤④「退后」之后自动出现；或点击「开始计时」前作为独立区块展示。

UI 形式：可折叠区块「今天留意这几件事」，**默认展开**，不阻断「开始计时」按钮。父母可关闭，下次重新打开同一活动卡时恢复展开（每次活动前均展示，不做已读记录）。

内容来源：`Activity.observeFocus[]`（每条含 `look` 行为描述 + `why` 发育意义）

```
今天留意这几件事  ▾ （点击收起）
  · 球消失后，他的眼睛/手有没有去找？
    → 知道消失的东西还在——客体永久性的核心信号
  · 他拿球时，会不会先看圆孔再对准放入？
    → 目的性行为出现——从随机探索进入有意图操作
  · 遇到放不进去，他有没有换角度再试？
    → 自主纠错能力的早期表现

[开始计时]
```

设计细节：
- `why` 字段以 `→` 缩进小字展示，视觉权重低于 `look`（行为事实优先）
- 2条为最少，3条为上限，避免父母记忆负担
- 文案要求：描述可观察的行为，不出现「应该」「要求」等词

---

**第2层：计时中观察锚点**

触发时机：点击「开始计时」后，全程显示于计时界面。

UI 形式：计时数字正下方，单行灰色小字（`color: var(--text-sec)`），不遮挡计时数字，不可关闭（但字体小，不干扰）。

内容来源：`Activity.observeAnchor`（单句，≤25字，第一人称视角的具体行为描述）

```
┌────────────────────────────────┐
│                                │
│         12:48                  │
│    ████████░░░░░░              │
│                                │
│  留意：球消失后，他有没有用手去找？  │
│                                │
│    [停止计时]                   │
│                                │
└────────────────────────────────┘
```

设计细节：
- observeAnchor 是 observeFocus 第1条的极简版，两者保持语义一致
- 不显示「今天留意」完整列表（避免分心），只保留最核心一句
- 不带发育意义解释（why），只保留行为事实（look）
- **计时结束后**：observeAnchor 原文以「？」拆分为填空问题，出现在记录表单中（M4 第3层衔接，见 M4 observeAnchor 填空引导说明）

**第3层：提交后即时数据反馈**

触发时机：M4 记录表单成功提交后立即出现，底部小卡片（非弹窗，不阻断返回），5秒后自动消失，或父母手动关闭。

内容逻辑：基于 `focusSec`、`ageMonths`、`initType` 三字段生成规则文案，无 AI 调用。

**专注时长解读**（主要内容，必选其一）：

| 条件 | 展示文案 |
|------|---------|
| `focusSec > typicalSec[ageMonths] * 1.5` | 「他今天专注了 X分Y秒，比这个月龄典型的约 Z 分钟长不少。」 |
| `focusSec < typicalSec[ageMonths] * 0.5` | 「他今天专注了 X分Y秒，比典型稍短——可以试试降难版本看看？」（降难字样可点击） |
| `0.5 ≤ focusSec/typicalSec ≤ 1.5` | 「他今天专注了 X分Y秒，在这个月龄的正常范围内。」 |
| `focusSec` 数据异常（< 30秒） | 不显示专注解读，避免误判（可能是手误） |

**自主行为追加**（可选附加，出现在上述文案后）：

| 条件 | 追加文案 |
|------|---------|
| `initType === 'self'` | 「而且是他自己过来拿的，主动探索的信号。」 |
| `initType === 'invited'` | （不追加，中性） |
| `initType === 'led'` | 「今天是带着他做的，偶尔这样没问题。」 |

**参考数据表**：`typicalSec[ageMonths]`（静态常量，来自分类方案 E.3 典型专注时长参考，单位：秒）

```js
const TYPICAL_FOCUS_SEC = {
  0: 60,   // 0-3月
  3: 90,   // 3-6月
  6: 150,  // 6-9月
  9: 210,  // 9-12月
  12: 300, // 12-18月
  18: 420, // 18-24月
  24: 600, // 2-3岁
}
// 使用时：取 floor(ageMonths / 3) * 3 对应的值
```

**第4层：活动卡阶段提示**

触发时机：每次打开活动详情 Modal 时，根据该活动的历史记录数计算当前阶段，渲染提示。

计算规则：
```js
const sessionCount = await DB.getByIndex('records', 'by-act', actId).length
const currentPhase = activity.phases.find(p =>
  sessionCount >= p.range[0] &&
  (p.range[1] === null || sessionCount <= p.range[1])
)
```

UI 形式：活动卡 Modal 顶部的 Tip Chip（绿色浅背景，`var(--primary-light)`），首次进入（sessionCount = 0）不显示（无历史参考）。

```
┌──────────────────────────────────────┐
│  ● 操作期提示（第3次活动）              │
│  他开始有意图了，今天留意：             │
│  他拿球时有没有看向圆孔？               │
│  哪怕只是瞄了一眼                      │
└──────────────────────────────────────┘
  ↓ 下方继续展示材料、步骤等活动详情
```

设计细节：
- Chip 右上角有关闭按钮（仅关闭本次，下次打开仍展示）
- 探索期提示（第1-2次）的语气为「放手观察」；操作期提示聚焦行为捕捉；熟练期提示引导父母关注创造性玩法
- phases[].label（探索期/操作期/熟练期）在 Chip 标题中展示，让父母建立阶段认知

---

**4层融合的完整用户旅程**：

```
打开活动详情 Modal
  ↓
[第4层] 顶部 Chip：当前阶段提示（sessionCount > 0 时）
  ↓
查看材料、步骤 → 展开「活动小贴士」
  ↓
[第1层] 今天留意这几件事（observeFocus，2-3条，展开态）
  ↓
点击「开始计时」→ 进入计时界面
  ↓
[第2层] 计时中底部显示 observeAnchor 锚点句
  ↓
活动结束 → 点击「停止计时」→ 弹出 M4 记录表单
  ↓
填写4字段 → 提交
  ↓
[第3层] 即时反馈卡（专注时长解读 + initType 追加，5秒自动消失）
  ↓
下次打开同一活动：循环，第4层 Chip 阶段随 sessionCount 自动升级
```

**首次使用渐进规则**（防止认知过载）：
- 第1层（开始前焦点）：首次进入活动时全部展示，但 `why` 字段默认折叠（只显示 `look`），用户点击才展开发育意义
- 第4层（阶段提示）：sessionCount = 0 时不显示，第1次完成记录后下次才开始出现
- 第2层和第3层：从第1次活动起始终存在，因为它们最简（单句/小卡片）

### M_GuideIntensity. 引导强度切换（v1.6新增）

**设计原则**：适配父母的蒙氏理念成熟度。新手父母需要详细引导，有经验父母需要轻量提示。通过一个设置开关，实现 UI 的差异化呈现。

**三个模式及含义**：

| 模式 | 目标用户 | 特点 |
|------|--------|------|

**各模式下的功能变化**：

#### 轻模式（Minimal）

### 8.3 PWA 更新机制（v1.8新增）

**策略**：Cache-First + `controllerchange` 事件通知，**完全离线安全**。

```
CACHE_NAME 变更（如 qiqi-v3 → qiqi-v4）
    ↓
浏览器后台下载新 SW（无网络时不触发，不影响离线使用）
    ↓
新 SW 激活（skipWaiting + clients.claim）
    ↓
触发 controllerchange 事件
    ↓
顶部显示提示条：「新版本已就绪  [刷新]」
    ↓
用户点击「刷新」→ location.reload()（使用新缓存）
```

**实现**（`js/app.js`）：
```js
navigator.serviceWorker.addEventListener('controllerchange', () => {
  // 显示顶部非强制刷新条，用户手动决定何时刷新
});
```

**升级缓存方法**：修改 `service-worker.js` 中的 `CACHE_NAME` 版本号，部署后浏览器自动完成后台更新。
